Step 1: Why do we even need Event Delegation?

Imagine this:

<ul>
  <li>Apple</li>
  <li>Mango</li>
  <li>Orange</li>
</ul>

You can add a click listener to every <li>:

const items = document.querySelectorAll("li");

items.forEach(item => {
    item.addEventListener("click", () => {
        console.log(item.textContent);
    });
});

This works, but what if:

you have 1000 <li> elements?
new <li> elements are added later?

The new elements won't have the listener.

Example:

const li = document.createElement("li");
li.textContent = "Banana";
document.querySelector("ul").appendChild(li);

Clicking Banana does nothing.

Event Delegation Solution

Instead of listening on every child, listen on the parent.

const ul = document.querySelector("ul");

ul.addEventListener("click", (e) => {
    console.log(e.target.textContent);
});

Now:

click Apple → Apple
click Mango → Mango
click Banana (added later) → Banana

because the event bubbles up.

How it works

When you click:

<li>Apple</li>

the event travels:

li
↑
ul
↑
body
↑
html
↑
document

This is called event bubbling.

The ul catches the event and can figure out which child was clicked.

The Magic: event.target
ul.addEventListener("click", (e) => {
    console.log(e.target);
});

If you click:

<li>Apple</li>

Output:

<li>Apple</li>

e.target = the actual element that started the event.

Real Example: Remove Todo Items

HTML:

<button id="add">Add</button>
<ul id="todos"></ul>

JS:

const btn = document.getElementById("add");
const ul = document.getElementById("todos");

let count = 1;

btn.addEventListener("click", () => {
    const li = document.createElement("li");
    li.textContent = `Todo ${count++}`;
    ul.appendChild(li);
});

ul.addEventListener("click", (e) => {
    if (e.target.tagName === "LI") {
        e.target.remove();
    }
});

No matter how many <li> elements you create, they all work.

Why tagName === "LI"?

Suppose your <li> contains:

<li>
    <span>Todo 1</span>
</li>

If you click the span:

e.target.tagName

becomes:

SPAN

not LI.

So sometimes we use:

e.target.closest("li")

which is even better.

ul.addEventListener("click", (e) => {
    const li = e.target.closest("li");

    if (li) {
        li.remove();
    }
});

This is the professional approach.

Another Common Use: Buttons
<div id="container">
    <button>1</button>
    <button>2</button>
    <button>3</button>
</div>
container.addEventListener("click", (e) => {
    if (e.target.matches("button")) {
        console.log(e.target.textContent);
    }
});

matches() checks if the clicked element matches a CSS selector.

Professional Pattern
parent.addEventListener("click", (e) => {
    const element = e.target.closest(".child");

    if (!element) return;

    // do something
});

You'll see this pattern everywhere in large applications.

When should you use Event Delegation?

✅ Dynamic elements

✅ Lots of similar elements

✅ Lists, tables, menus, todo apps

✅ Better performance

When should you NOT use it?

❌ Events that don't bubble:

focus
blur
mouseenter
mouseleave

For these, delegation usually won't work directly.

Problem 1: Nested Elements

Suppose you have:

<ul id="users">
    <li>
        <span>Ali</span>
    </li>

    <li>
        <span>Ahmed</span>
    </li>
</ul>

You write:

users.addEventListener("click", (e) => {
    if (e.target.tagName === "LI") {
        e.target.style.color = "red";
    }
});
Problem:

Clicking the text (span) does nothing because:

e.target.tagName
// "SPAN"

not "LI".

Solution: closest()
users.addEventListener("click", (e) => {
    const li = e.target.closest("li");

    if (!li) return;

    li.style.color = "red";
});

Now it works even if you click the span.

Why closest() is amazing

It starts from the clicked element and moves upward:

span
↑
li
↑
ul
↑
body

until it finds a matching element.

<div id="products">

Instead of:

buyBtn.addEventListener(...)

One listener handles everything.

Problem 3: Clicking Outside the Intended Area

Suppose:

<div id="container">
    <button>One</button>
    <button>Two</button>
</div>

You do:

container.addEventListener("click", (e) => {
    console.log(e.target.textContent);
});

Clicking the empty space inside the div gives:

""

because:

e.target === container
Better:
container.addEventListener("click", (e) => {
    const btn = e.target.closest("button");

    if (!btn) return;

    console.log(btn.textContent);
});
target vs currentTarget

This is extremely important.

container.addEventListener("click", (e) => {
    console.log(e.target);
    console.log(e.currentTarget);
});

Suppose:

<div id="container">
    <button>Click</button>
</div>

Click button:

e.target
→ button

e.currentTarget
→ div
Rule:
target = who started the event
currentTarget = who owns the listener
Professional Pattern

You'll see this everywhere:

parent.addEventListener("click", (e) => {
    const target = e.target.closest(".item");

    if (!target || !parent.contains(target))
        return;

    // do something
});

The contains() check prevents accidentally selecting elements outside your parent.

Example: Dynamic Cards
<div id="cards"></div>
cards.addEventListener("click", (e) => {
    const btn = e.target.closest(".delete-btn");

    if (!btn) return;

    btn.closest(".card").remove();
});

Works for:

existing cards ✅
newly created cards ✅
thousands of cards ✅

The Event Journey

When you click an element, the event goes through three phases:

document
↓
html
↓
body
↓
div
↓
button   ← Target
↑
div
↑
body
↑
html
↑
document

The phases are:

Capturing phase (going down)
Target phase (reaches the clicked element)
Bubbling phase (going back up)

Most of the time, event delegation uses bubbling.

Example
<div id="parent">
    <button id="btn">Click</button>
</div>
parent.addEventListener("click", () => {
    console.log("Parent");
});

btn.addEventListener("click", () => {
    console.log("Button");
});

Click the button:

Button
Parent

because the click bubbles upward.

Capturing

You can listen during the capturing phase:

parent.addEventListener(
    "click",
    () => {
        console.log("Parent Capture");
    },
    true
);

or

parent.addEventListener("click", handler, {
    capture: true
});

Now clicking the button gives:

Parent Capture
Button
Parent
Why does Event Delegation work?

Because of this:

button clicked
↓
event bubbles
↓
parent receives it
↓
parent checks e.target

Without bubbling, event delegation wouldn't exist.

stopPropagation()

Suppose:

btn.addEventListener("click", (e) => {
    e.stopPropagation();
    console.log("Button");
});

parent.addEventListener("click", () => {
    console.log("Parent");
});

Click button:

Button

The event never reaches the parent.

Why is this important?

Your delegated listener may suddenly stop working because some child does:

e.stopPropagation();

This is a common bug in large apps.

stopImmediatePropagation()
btn.addEventListener("click", (e) => {
    e.stopImmediatePropagation();
});

This stops:

bubbling
other listeners on the same element.
Event Delegation Utility Function

Professionals often create helpers.

function delegate(parent, eventType, selector, handler) {
    parent.addEventListener(eventType, (e) => {
        const target = e.target.closest(selector);

        if (!target || !parent.contains(target))
            return;

        handler(e, target);
    });
}

Usage:

delegate(
    document,
    "click",
    ".delete-btn",
    (e, btn) => {
        btn.closest(".card").remove();
    }
);

Pretty clean, right?

Multiple Actions Pattern

HTML:

<button data-action="save">Save</button>
<button data-action="delete">Delete</button>
<button data-action="edit">Edit</button>

JS:

document.addEventListener("click", (e) => {
    const btn = e.target.closest("[data-action]");

    if (!btn) return;

    const action = btn.dataset.action;

    if (action === "save") {
        console.log("Saving...");
    }

    if (action === "delete") {
        console.log("Deleting...");
    }

    if (action === "edit") {
        console.log("Editing...");
    }
});

This pattern is used everywhere.
